# Which-key transient menus + git command groups

Reorganize the keymap into a magit/neogit-style two-level transient system with a
which-key bottom panel. Phase 1 (this plan) builds the framework + bottom panel +
reorganizes EXISTING actions into transients (commit, branch, log) + prefix-only
help + conflict enforcement + full config. Push/pull/fetch (new network backend)
are Phase 2.

Research: which-key.nvim timing/layout, magit/neogit canonical groupings — see the
agent findings in this session.

## Conflict-avoidance rule (the design constraint)

A key at a given level is EITHER a leaf (immediate action) OR a prefix (opens a
transient) — never both. Top-level: a small set of direct keys (j/k, q, ?, tab,
nav, refresh, the review actions) plus prefix keys that open transients. Inside a
transient every key is a pure leaf (or a deeper transient under a distinct key).
Disambiguate variants by case (lower = dwim, upper = explicit), magit-style.

## Transient model

- `Transient { title, groups: Vec<TransientGroup{ heading, entries: Vec<(chord, Action, label)> }> }`.
  Pressing a top-level prefix enters a transient state; the next key resolves
  against that transient's entries; Esc aborts; Backspace pops one level.
- This generalizes the current two-key sequences (cc, ll) and the ad-hoc branch
  popup (Modal::Branch) into one mechanism. Remove the bespoke Modal::Branch /
  two-key keymap handling in favor of transients.
- Keep it pure/testable: a `Transient::resolve(key) -> Resolved{Action|Deeper|None}`.

## Which-key bottom panel

- On entering a transient, arm a timer; after ~200ms with no resolving key, draw a
  bottom panel listing the transient's groups as columns of `key  label` (group
  headings as column headers). A fast second keystroke resolves before the panel
  shows (never flashes for fast typists). The panel is a transient overlay at the
  bottom (reserve rows via Layout), sized to terminal width: column width =
  max(key)+sep+max(label), `floor((w+spacing)/(col+spacing))` columns, wrap to
  bands, cap height with scroll.
- Esc closes; Backspace up one level; an unknown key beeps (status-bar) and stays
  (or closes — pick: closes, matching neogit).
- The TUI tick (250ms) drives the reveal timer; store `transient_opened_at` as a
  tick counter (no wall-clock in render).

## Top-level keys (status screen) + transients

Direct (leaves): `j/k` move, `<tab>` fold, `s/u/S/U` stage, `x` discard, `<cr>`
open, `<c-r>` refresh, `{`/`}` hunks, `q` quit, `?` help, `g g`/`G`.
Prefixes → transients (neogit scheme):
- `c` Commit: `c` commit, `e` extend, `a` amend, `w` reword.
- `b` Branch: `b` switch (checkout existing — branch-list picker), `c` create+
  checkout, `n` create, `m` rename (defer if no backend), `D` delete.
- `l` Log: `l` current branch log (opens the log screen).
- (Phase 2 will add `P` Push, `p` Pull, `f` Fetch, `Z` Stash, `X` Reset.)

Diff screen keeps its review leaves (c comment, V select, r/R, v, y/Y, e, c-n/c-p,
tab focus) — note `c` is a comment leaf in the diff context and a transient prefix
in the status context; that's fine (different contexts, no conflict within a
context). Keep contexts separate.

## Help header = prefixes only

The hint line shows only the top-level keys/prefixes (e.g. `c commit  b branch
l log  s stage  ? help`), not every sub-command. The full sub-commands appear in
the which-key panel when you hit the prefix, and in the `?` help popup (which
already lists the active keymap — extend it to show transient groups).

## Configurability + conflict enforcement (HARD requirement)

- Every binding (top-level AND transient sub-keys) is overridable via the existing
  `[keys.*]` TOML config. Extend the config schema so transient sub-keys are
  addressable (e.g. `[keys.commit] amend = "a"`, `[keys.branch] delete = "D"`).
- On load, ENFORCE no conflicts: within any one level (top-level of a context, or
  one transient), no two actions share a chord, and no chord is both a leaf and a
  prefix. Emit a config warning (existing warning flow) for each conflict and fall
  back to the default for the conflicting override. Add a startup self-check over
  the BUILT-IN defaults too (a test asserting the defaults are conflict-free per
  level).

## Tests

- Transient resolve unit tests; conflict-detection tests (overlapping override
  warns + falls back; leaf/prefix clash detected); which-key panel layout (column
  packing math) unit test; the reveal-timer gating (fast key resolves before
  panel; slow shows panel) via the tick model; snapshots of the which-key panel
  for the commit and branch transients; help-header-shows-prefixes-only snapshot.
- Keep all existing review/nav behavior green.

## Out of scope (Phase 2)

Push/pull/fetch/stash/reset/rebase (need network + git2 credential callbacks in
the Vcs trait); nested >2-level transients; mouse menu clicks.
